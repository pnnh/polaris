# PostgreSQL 版本兼容性处理说明

## 概述

`export_tables.bash` 和 `import_tables.py` 脚本已经增强了对不同 PostgreSQL 版本间数据迁移的支持。

## 版本兼容性处理机制

### 1. 导出脚本增强 (export_tables.bash)

**新增选项**：
```bash
pg_dump ... \
  --no-owner \              # 不导出对象所有者
  --no-privileges \         # 不导出权限
  --quote-all-identifiers \ # 引用所有标识符
  --encoding=UTF8           # UTF-8 编码
```

**作用**：
- `--no-owner`: 避免不同环境用户名差异导致的导入失败
- `--no-privileges`: 跳过权限相关的 GRANT/REVOKE 语句，避免权限系统版本差异
- `--quote-all-identifiers`: 确保所有标识符（表名、列名等）都被引号包围，避免新旧版本关键字冲突
- `--encoding=UTF8`: 统一使用 UTF-8 编码，确保字符集兼容

### 2. 导入脚本增强 (import_tables.py)

**新增功能**：

#### a) 自动版本检测
```python
# 检测目标数据库版本
target_version = get_pg_version()

# 从 SQL 文件提取源版本
source_version = extract_source_version_from_sql(sql_file)
```

#### b) 版本兼容性检查
- 自动对比源版本和目标版本
- 识别升级、降级或同版本迁移
- 针对风险场景显示警告

#### c) 交互式确认
- 检测到版本不兼容时提示用户
- 允许用户选择是否继续

## 兼容性场景分析

### ✅ 场景1：向前兼容（旧→新，如 16 → 18）

**兼容性**: 良好  
**风险等级**: 低到中等

**特点**：
- PostgreSQL 通常保持良好的向前兼容性
- pg_dump 生成的 SQL 可以在新版本上执行
- 新版本可以理解旧版本的语法

**注意事项**：
```
✓ 大部分情况下无缝迁移
⚠️ 某些弃用功能可能已移除
⚠️ 数据类型默认行为可能改变
⚠️ 建议先在测试环境验证
```

**脚本行为**：
- 检测到跨版本升级（跨度>2）时显示提示
- 不阻止操作，但建议用户留意

### ⚠️ 场景2：向后兼容（新→旧，如 18 → 16）

**兼容性**: 风险较高  
**风险等级**: 高

**特点**：
- 新版本特性在旧版本中不可用
- 可能出现类型、函数、语法不兼容

**常见问题**：
```
✗ 新增的数据类型（如 PostgreSQL 14+ 的新类型）
✗ 新增的函数和操作符
✗ 新的 SQL 语法特性
✗ 系统表结构变化
```

**脚本行为**：
- 显示警告信息
- 要求用户确认是否继续
- 建议在测试环境先验证

**示例提示**：
```
============================================================
版本降级警告：从 PostgreSQL 18 导入到 16
  - 新版本特性可能不兼容
  - 强烈建议先在测试环境验证
============================================================
是否继续导入? (y/n): 
```

### ✅ 场景3：同主版本（如 16.11 → 16.12）

**兼容性**: 最佳  
**风险等级**: 极低

**特点**：
- 次版本更新通常是 bug 修复和安全补丁
- 完全向前向后兼容

**脚本行为**：
- 检测到同主版本，显示兼容性良好
- 无需特殊处理

## 故障排除

### 问题1: 导入时报"权限不足"
**原因**: 目标数据库用户权限不同  
**解决**: 使用 `--no-owner --no-privileges` 导出（已默认启用）

### 问题2: 导入时报"类型不存在"
**原因**: 向后兼容问题，新版本类型在旧版本不存在  
**解决**: 
- 检查是否必须降级
- 考虑在旧版本中手动创建缺失类型
- 或修改导出文件移除新类型

### 问题3: 导入时报"语法错误"
**原因**: SQL 语法在版本间有差异  
**解决**: 
- 使用 `--quote-all-identifiers` 导出（已默认启用）
- 检查是否使用了特定版本的语法特性

## 最佳实践

### ✅ 推荐做法
1. **同主版本迁移**: 优先选择 16.x → 16.y
2. **逐步升级**: 跨大版本时分步进行（16 → 17 → 18）
3. **测试先行**: 先在测试环境完整验证
4. **备份数据**: 导入前备份目标数据库
5. **查阅文档**: 阅读版本发布说明了解破坏性变更

### ⚠️ 避免做法
1. **跨主版本降级**: 除非必要，避免从新版本降级到旧版本
2. **直接生产**: 不要在生产环境直接测试跨版本导入
3. **忽略警告**: 脚本显示警告时应认真对待
